### Reids

#### 1.是什么？

https://www.jianshu.com/p/56999f2b8e3b

**定义：**是一个非关系型的数据库，数据都是保存在内存当中。

NoSQL技术：基于内存的数据库。

常见的有redis和MongoDB。

> **关系与非关系数据库：**
>
> **关系：**二维表以及表之间的关系，我们从二维表中能看出关系趋势等信息
>
> **好处：**贴近现实逻辑、使用方便易于维护，但是由于保存在磁盘上，遇到高并发会使其you奔溃的风险
>
> 
>
> **非关系：**以键值对存储
>
> **好处：**根据需要添加字段，仅仅根据id就可以查出需要的value，不像关系型数据库还得进行多表查询
>
> **缺点：**不能存储海量数据

#### 2.特点

**优势：**

可以将数据持久化到硬盘

1. **速度快，每秒十几万次的读写操作**
2. **支持集群、主从同步等配置**，支持事务，保证了并发情况下数据的安全

​    3. [**数据结构**](http://lib.csdn.net/base/datastructure)**丰富：除了支持string类型的value外还支持string、hash、set、sortedset、list等数据结构。**

**缺点：**



#### **3.使用背景**

我们一般都是使用数据库进行数据存储，数据被保存在磁盘上。在一般情况下这个是没有问题的，但是如果遇到了高并发，在短时间内上万的访问量要求数据库进行读写操作，而磁盘的读写速度是比较慢的，这个时候数据库就无法承受这么大的激增的访问量，就会导致数据库系统瘫痪。这时通过使用将数据保存在内存当中的redis数据库可以很好的解决这个问题。

**redis有两种用途——缓存数据高速读取;**

在对数据库的访问中，一般情况下读操作要多于写操作，如果从数据库中查找记录这是一个比较慢的过程，但是如果我们将数据直接放在redis也就是内存中的时候，这样就会提高速度，但是我们不能将所有的数据都放在内存当中会增加成本。

所以一般情况下我们存在redis中的数据要具备以下条件：

1. 读操作多于写操作
2. 数据大小较小
3. 数据常用

一般我们会存储用户的登录信息等。

#### 4.5种redis中的数据结构（value的类型）

**1.字符串：保存的是字符串、整数或者浮点数**

命令：get、set、del

微博粉丝数目

实现：redis自己创建了一个SDS的数据结构。

```
struct sdshdr{
     //记录buf数组中已使用字节的数量
     //等于 SDS 保存字符串的长度
     int len;
     //记录 buf 数组中未使用字节的数量
     int free;
     //字节数组，用于保存字符串
     char buf[];
}
```

**2.链表：**一个链表，链表的每个节点都包含了一个字符串（可以存储多个相同的字符串）

命令：LPUSH/RPUSH分别将元素推入列表的左端和右端

LPOP  RPOP分别从列表的左端和右端弹出元素

LINDEX获取列表在给定位置上的一个元素

LRANGE获取列表在给定范围上所有元素//使用0为起始-1为结束的范围索引可以提取出列表包含的所有元素

实现：

```
typedef struct list{
     //表头节点
     listNode *head;
     //表尾节点
     listNode *tail;
     //链表所包含的节点数量
     unsigned long len;
     //节点值复制函数
     void (*free) (void *ptr);
     //节点值释放函数
     void (*free) (void *ptr);
     //节点值对比函数
     int (*match) (void *ptr,void *key);
}list;
```

根据源码可以得知：可以直接获得头、尾结点；常数时间复杂度获得链表长度；是双向链表

**3.字典：**和列表一样可以存储多个字符串，但是每个字符串都**不能重复**，因为使用了**散列表**（**这些散列表只有键而没有相关的值**）

实现：在`数组+链表`的基础上，进行了一些rehash优化等。

1. Reids的Hash采用链地址法来处理冲突，然后它没有使用红黑树优化。
2. 哈希表节点采用单链表结构。
3. rehash优化。

操作：SADD添加  

SREM移除

SISMEMBER检查一个元素是否已经存在于集合中

SISMEMBERS获取集合包含的所有元素

**4.散列：**

可以存储多个键值对之间的映射。

散列中存储的值可以是字符串也可以是数字值

应用：可以用来保存一个对象，这样可以直接修改对象中的字段的值。

note：多个键值对指的是对象中的属性和其对应的值。

命令：

HSET添加散列对

HGET根据键获得值

HGETALL获取所有键值对

HDEL移除键值对（根据给定键）



**5.有序集合：**

和散列一样存储键值对，不同的是里面的键叫做成员，值叫做分值，而且分值只能是浮点数

而且是唯一一个可以根据成员访问分值，也可以根绝分值访问成员的结构

例如直播了礼物榜等

命令：

ZADD将一个给定分值的成员添加到有序集合中

ZRANGE根据元素所处的位置，从有序结合里面获取多个元素

ZREM删除

**6.跳表skiplist：**

1、一个跳表应该有几个层（level）组成；
通常是10-20层，leveldb中默认为12层。

2、跳表的第0层包含所有的元素；
且节点值是有序的。

3、每一层都是一个有序的链表；
层数越高应越稀疏，这样在高层次中能'跳过’许多的不符合条件的数据。

4、如果元素x出现在第i层，则所有比i小的层都包含x；

5、每个节点包含key及其对应的value和一个指向该节点第n层的下个节点的指针数组
x->next[level]表示第level层的x的下一个节点

> Hash表：插入、查找最快，为O(1)；如使用链表实现则可实现无锁；数据有序化需要显式的排序操作。
> 红黑树：插入、查找为O(logn)，但常数项较小；无锁实现的复杂性很高，一般需要加锁；数据天然有序。
> SkipList：插入、查找为O(logn)，但常数项比红黑树要大；底层结构为链表，可无锁实现；数据天然有序。

#### 5.持久化（让用户将自己的数据存储到硬盘上）

**快照持久化：**通过创建快照来获得存储在内存里面的数据在每个时刻的副本

保存某一时刻的数据副本到硬盘。

缺点：如果系统奔溃的话，将丢失最近一次快照之后的所有更改的数据。

原理：**BGSAVE**指令：执行该指令就会产生一个子进程去生产副本

**SAVE 60  10000**意思是在60秒之内如果出现了10000次的写入时就会自动触发BGSAVE操作

**AOF:**

保存写命令到AOF文件，然后redis会执行AOF文件中的写入指令来进行数据的更新。

文件同步：将文件写入硬盘的时候，会先将文件存储在缓冲区，然后操作系统会在将来的某个时间将缓冲区的数据写入硬盘（这个动作是不确定的）。用户可以命令操作系统将文件同步到硬盘，这会阻塞直到文件写入硬盘。  

同步的次数：每次写入都同步、一秒同步一次、不同步。一般选择一秒一次。

AOF文件体积过大问题，而且体积过大会导致执行AOF文件时间过长的问题。

需要对其进行压缩，通过重写AOF文件对其中冗余的写入操作进行清除。

数据的备份、持久化做完了，我们如何从这些持久化文件中恢复数据呢？如果一台服务器上有既有RDB文件，又有AOF文件，该加载谁呢？

一般是先加载aof文件，因为aof文件顶多损耗1秒的数据

#### 6.过期策略

https://segmentfault.com/a/1190000016810883?utm_source=tag-newest

Redis 所有的数据结构都可以设置过期时间，时间一到，就会自动删除。

redis会将设置了过期时间段数据放在一个字典里，然后会**定时遍历**，来删除到期的key。

还有一种**懒惰机制**，当客户端访问这个key的时候，会检查过期时间，如果过期就立马回收掉。前者是集中处理，后者是零散处理

定时遍历策略：贪心算法

从过期字典中随机 20 个 key；

删除这 20 个 key 中已经过期的 key；

如果过期的 key 比率超过 1/4，那就重复步骤 1；

#### 7.内存淘汰机制

内存的淘汰机制的初衷是为了更好地使用内存，用一定的缓存miss来换取内存的使用效率。

通过配置redis.conf中的maxmemory这个值来开启内存淘汰功能

Redis提供了下面几种淘汰策略供用户选择，其中默认的策略为noeviction策略：

1. ​      noeviction：当内存使用达到阈值的时候，所有引起申请内存的命令会报错。
2. ​      allkeys-lru：在主键空间中，优先移除最近未使用的key。
3. ​      volatile-lru：在设置了过期时间的键空间中，优先移除最近未使用的key。
4. ​      allkeys-random：在主键空间中，随机移除某个key。
5. ​      volatile-random：在设置了过期时间的键空间中，随机移除某个key。
6. ​      volatile-ttl：在设置了过期时间的键空间中，具有更早过期时间的key优先移除。

#### 8.事务机制

redis事务是命令的集合，要么全部被执行，要么全不被执行。

一个redis事务从开始到执行经过如下三个阶段：

https://redisbook.readthedocs.io/en/latest/feature/transaction.html

1. 开始事务，`MULTI`

   将客户端的 `REDIS_MULTI` 选项打开， 让客户端从非事务状态切换到事务状态。

2. 命令入队，

   当客户端处于非事务状态下时， 所有发送给服务器端的命令都会立即被服务器执行；但是， 当客户端进入事务状态之后， 服务器在收到来自客户端的命令时， 不会立即执行命令， 而是将这些命令全部放进一个事务队列里， 然后返回 `QUEUED` ， 表示命令已入队。

3. 执行事务，`EXEC `，这是命令才开始会被依次被执行

   如果客户端正处于事务状态， 那么当 [EXEC](http://redis.readthedocs.org/en/latest/transaction/exec.html#exec) 命令执行时， 服务器根据客户端所保存的事务队列， 以先进先出（FIFO）的方式执行事务队列中的命令： 最先入队的命令最先执行， 而最后入队的命令最后执行。

事务错误处理：

1. 语法错误：会导致所有命令都不会被执行
2. 运行出错：这种命令在执行之前是无法被发现的，所以其他的命令还是会被执行

> **Redis中的事务并没有关系型数据库中的事务回滚(rollback)功能，因此使用者必须自己收拾剩下的烂摊子。**不过由于Redis不支持事务回滚功能，这也使得Redis的事务简洁快速。

WATCH命令：

监控一个键不被修改直到EXEC执行完之后。

WATCH命令可以监控一个或多个键，一旦其中有一个键被修改（或删除），之后的事务就不会执行，监控一直持续到EXEC命令（事务中的命令是在EXEC之后才执行的，EXEC命令执行完之后被监控的键会自动被UNWATCH）

#### 8.redis就真的那么完美无缺吗？

1. 虽然redis本身有key过期策略，但是还是需要提前预估和节约内存。如果内存增长过快，需要定期删除数据。
2. 如果进行完整重同步，由于需要生成rdb文件，并进行传输，会占用主机的CPU，并会消耗现网的带宽。redis是一个cpu密集型服务器。
3. 修改配置文件，进行重启，将硬盘中的数据加载进内存，时间比较久。在这个过程中，redis不能提供服务。
